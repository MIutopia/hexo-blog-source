## 什么是 Vibe Coding？

Vibe Coding（氛围编程）是 AI 研究员 Andrej Karpathy 提出的概念——**用自然语言描述需求，让 AI 替你写代码**。开发者不再逐行敲键盘，而是像产品经理一样描述想法，然后审查 AI 产出的结果。

```
传统编程：想清楚 → 写代码 → 调试 → 修复
Vibe Coding：描述需求 → AI 写代码 → 审查 → 调整描述
```

这听起来有点像"告别编程"，但实际上它改变的只是**编码方式**，而不是**工程思维**。

## Vibe Coding 不是"不用动脑"

很多人误解 Vibe Coding 就是躺着等 AI 干活。**不是的。** 真正有效的 Vibe Coding 需要：

- **清晰的描述能力** — 能把模糊想法转化成准确的文字指令
- **代码审查能力** — 能判断 AI 写的代码对不对、好不好
- **调试引导能力** — 当 AI 写出 bug 时，能引导它修复
- **架构决策能力** — 知道整体怎么设计，只是把实现细节交给 AI

**你的角色从"打字员"变成了"技术主管"**。

## 怎么开始 Vibe Coding？

### 1. 选对工具

- **Claude Code** — 终端里的 AI 编程助手，能读写文件、执行命令、理解项目上下文
- **Cursor / GitHub Copilot** — IDE 内嵌的 AI 补全和对话

### 2. 写好项目规范

创建一个 `CLAUDE.md` 文件放在项目根目录，告诉 AI 你的项目规则：

```markdown
# 项目规范

## 技术栈
- Hexo 博客框架
- Fluid 主题
- 部署在 Vercel

## 写作规范
- 文章用 Markdown 格式
- frontmatter 必须包含 title、date、categories、tags
- 日期格式 YYYY-MM-DD HH:MM:SS
- 代码块标注语言类型（如 ```bash、```yaml）

## 约束
- 不要修改 _config.yml 除非明确要求
- 不要删除已有文件
- 提交信息格式：type: description
```

有了这份规范，AI 就知道你的项目该怎么干活了。

### 3. 用自然语言下达指令

**不好的指令：**
> "帮我改一下文章"

**好的指令：**
> "source/_posts/ 目录下的旧文章格式不统一：
> 1. 把标题改为 `标题 - 副标题` 格式
> 2. 补全缺失的 categories
> 3. 把代码块加上语言标记（yaml、bash、dockerfile）
> 4. HTML 实体转回正常字符
> 每改完一篇就提交一次"

越具体，AI 输出质量越高。

### 4. 审查 + 迭代

AI 写完代码后，你需要检查：

- 逻辑是否正确？
- 风格是否符合项目规范？
- 有没有安全隐患？

如果不对，**不要手动改**，而是告诉 AI 哪里不对，让它自己修：

> "第三段配置有误，Nginx 的 upstream 应该指向 8080 端口而不是 80"

## Vibe Coding 的适用场景

### 适合 ✅

- 写脚本和工具函数
- 批量处理文件（改名、格式化、迁移）
- 生成模板代码
- 写自动化测试
- 配置文件和文档
- 重构代码

### 不适合 ❌

- 需要深度领域知识的复杂业务逻辑
- 安全性要求极高的核心系统
- 需要人类审美判断的设计工作
- 全新的、无先例的技术方案

## 我的 Vibe Coding 工作流

以我管理这个博客为例：

```
需求：写一篇 Vibe Coding 的文章
  ↓
Claude Code 读取项目规范（CLAUDE.md）
  ↓
按规范格式创建 Markdown 文件
  ↓
写入文章内容（按约定风格）
  ↓
本地 hexo generate 验证
  ↓
git commit + git push
  ↓
Vercel 自动部署上线
```

全程我没有手写过一行代码，只是描述需求、审查内容、确认发布。

## Vibe Coding 的未来

AI 编程能力在以惊人的速度进化。今天的 Vibe Coding 可能还停留在"写函数、改文件"的阶段，但很快会延伸到：

- **跨项目管理** — AI 同时维护多个代码仓库
- **自动修复 issue** — GitHub Issue 自动分析并提 PR
- **架构设计辅助** — 用自然语言讨论架构方案，AI 生成实现

## 总结

Vibe Coding 不是"编程的终结"，而是"编程的进化"——**从手动编码到意图驱动**。

真正值钱的能力不再是"敲键盘的速度"，而是：
1. 把复杂问题拆解成清晰指令的能力
2. 判断 AI 输出质量的能力
3. 引导 AI 走向正确方向的能力

如果你还在犹豫要不要尝试，我的建议是：**从一个小的、具体的任务开始**。比如让 AI 帮你批量格式化文件，或者写一个简单的脚本。体验一下"说句话就出代码"的感觉，你会回不去的。
