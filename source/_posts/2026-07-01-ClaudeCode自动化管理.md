## 前言

搭好 Claude Code + GitHub + Vercel 的环境后，你可能已经体验过"说句话就能写代码"的快感了。但这只是冰山一角。

Claude Code 真正的威力在于**自动化**——它不只是帮你写一段代码，而是能理解整个项目、批量执行操作、按你的规范自动管理文件。这篇文章会带你从"让 AI 写代码"进阶到"让 AI 管理项目"。

## Claude Code 能做什么，不能做什么

先搞清楚能力边界，避免期望过高或过低。

| 擅长 | 不擅长 |
|------|--------|
| 读写项目文件、修改配置 | 需要图形界面操作的任务（如拖拽 UI） |
| 执行命令行操作（git、npm、docker 等） | 实时交互类操作（如调试断点） |
| 理解项目上下文，按规范生成代码 | 超出训练数据范围的极新框架 |
| 批量处理重复性任务 | 需要人类审美判断的设计工作 |
| 多步骤复杂流程的编排 | 需要实时网络浏览的场景（部分受限） |

简单说：**凡是能在终端里完成的事，Claude Code 基本都能帮你做**。

## 项目指令文件：给 AI 队友写"入职手册"

Claude Code 有一个核心特性——它会自动读取项目根目录的 `CLAUDE.md` 文件。这个文件就是你的项目规范，告诉 Claude Code 你的项目是什么、该怎么干活。

### 一个完整的 CLAUDE.md 示例

```markdown
# CLAUDE.md

## 项目信息
Hexo 个人博客项目，部署在 Vercel 上。
源码仓库：MIutopia/hexo-blog-source

## 目录结构
- source/_posts/    文章目录（Markdown 格式）
- source/images/    图片资源
- themes/fluid/     主题文件

## 文章规范
- 每篇文章必须包含 YAML frontmatter：title、date、categories、tags
- 日期格式：YYYY-MM-DD HH:MM:SS
- 代码块必须标注语言类型（如 ```bash）
- 使用中文写作，技术术语保留英文原文

## 工作流
- 修改完成后用 git add . && git commit && git push 推送
- 提交信息格式：type: description
  - feat: 新功能/新文章
  - fix: 修复问题
  - docs: 文档更新
  - refactor: 重构

## 注意事项
- 不要修改 _config.yml 除非明确要求
- 不要删除任何已有文件
- 图片文件不要通过 Git 管理（太大）
```

有了这份"入职手册"，Claude Code 就知道：

- 文章该写在哪里、什么格式
- 改完代码该怎么提交
- 哪些文件不能碰

### 让指令文件生效

不需要任何额外配置，只要文件放在项目根目录，Claude Code 每次启动时会自动读取。你可以随时修改它，下次对话就会生效。

> ⚠️ 如果你的项目比较复杂，还可以创建 `AGENTS.md`（多 Agent 协作规范）或 `SOUL.md`（语气和风格定义），Claude Code 同样会自动识别。

## 实战：用 Claude Code 管理 Hexo 博客

下面用我的博客项目做例子，展示 Claude Code 在实际项目管理中的几种用法。

### 场景一：批量改写文章风格

博客里有十几篇从旧网站迁移过来的文章，格式混乱、代码块没有语言标记、还混着 HTML 实体。手动改太痛苦了。

```
你：source/_posts/ 目录下除了 2026-06-30 那篇，其他文章都是从旧网站
    迁移过来的，格式有问题。请帮我逐篇改写，统一成最新那篇的风格：
    YAML frontmatter → 前言 → 正文分节 → 常见问题 → 结尾。
    代码块加上语言标记，HTML 实体转回正常字符。
```

Claude Code 会：
1. 先读取参考文章（2026-06-30 那篇）学习风格
2. 逐篇读取旧文章
3. 按新风格重写
4. 保存回原文件

十几篇文章，几分钟就改完了。

### 场景二：批量分类和打标签

```
你：帮我看一下 source/_posts/ 下所有文章的 categories 和 tags，
    如果分类不合理就调整一下。Docker 相关的文章 categories 改成"容器"，
    Kubernetes 和 OpenStack 的改成"云原生"。
```

Claude Code 会读取每篇文章的 frontmatter，判断内容主题，然后批量修改。

### 场景三：生成文章摘要和 SEO 优化

```
你：给每篇文章在 frontmatter 里加一个 description 字段，
    用一句话概括文章内容，控制在 150 字以内，用于 SEO。
```

### 场景四：检查部署状态

```
你：帮我检查一下 Vercel 上最新的部署状态，有没有报错。
```

Claude Code 可以调用 Vercel 的 API 或 CLI 来检查部署日志。

## 自动化工作流设计

把上面的场景串起来，就是一个完整的自动化工作流：

```
你描述需求（自然语言）
       ↓
Claude Code 读取 CLAUDE.md 了解项目规范
       ↓
Claude Code 执行操作（写代码/改文件/跑命令）
       ↓
Claude Code 本地验证（检查语法、运行测试）
       ↓
git add → git commit → git push
       ↓
Vercel 检测到推送，自动构建部署
       ↓
部署完成，你收到通知
```

整个过程你只需要做第一步和最后一步的确认，中间全是自动的。

### 关键：让 Claude Code 自己决定提交时机

在 CLAUDE.md 里写清楚规则后，你可以直接说：

```
你：改完这篇文章后直接推送到 GitHub。
```

Claude Code 会自己执行 `git add`、`git commit`、`git push`，不需要你一步步教。

## 进阶玩法

### 定时任务自动巡检

Claude Code 支持定时任务功能。比如你可以设置每天早上自动检查：

```
每天早上 9 点，检查 Vercel 部署状态，如果有失败的就通知我。
```

### 多项目并行管理

如果你同时维护多个项目，每个项目根目录放一份 CLAUDE.md，Claude Code 切换到不同目录时会自动加载对应的规范。

```
你：切到 my-blog 项目，帮我检查一下有没有待推送的改动。
你：切到 portfolio 项目，把首页的联系方式更新一下。
```

### 用 Memory 保持上下文连贯

Claude Code 的 Memory 功能可以记住跨会话的信息。比如你告诉它：

```
你：记住一下，我的博客域名是 cloudetopia.top，部署在 Vercel 上。
```

以后每次对话它都知道这个信息，不需要你重复说明。

## 效率对比

同样一批任务，手动操作和 Claude Code 操作的对比：

| 任务 | 手动操作 | Claude Code |
|------|---------|-------------|
| 改写 14 篇文章格式 | 2-3 小时 | 10 分钟 |
| 批量调整分类标签 | 30 分钟 | 2 分钟 |
| 检查部署状态 | 打开网页 → 登录 → 查看 | 一句话 |
| 新增一篇博客文章 | 写内容 + 手动加 frontmatter + git 操作 | 描述需求，全自动 |
| 修复一个配置错误 | 查文档 + 改文件 + 测试 + 提交 | 描述问题，自动完成 |

效率提升不是线性的——**越重复、越机械的任务，AI 的优势越明显**。

## 常见问题与踩坑记录

❌ **Claude Code 不遵守 CLAUDE.md 里的规范**

检查文件是否在**项目根目录**（和 `.git` 同级），文件名必须是大写的 `CLAUDE.md`。如果项目结构复杂，可以在子目录也放一份。

❌ **批量操作时 Claude Code 改错了文件**

这是 AI 的常见问题。建议先让它在**一个文件上试改**，确认效果满意后再说"其他文件也按这个方式改"。Git 的存在就是为了防止这种意外——改错了可以回滚（下一篇详细讲）。

❌ **Claude Code 执行 git push 失败**

通常是认证问题。确认 Git 的 remote URL 正确（`git remote -v` 检查），以及你已经配置了 Personal Access Token 或 SSH Key。

❌ **Vercel 部署成功但页面没更新**

可能是浏览器缓存。尝试强制刷新（Ctrl+Shift+R），或者在 Vercel 控制台手动触发一次 Redeploy。

❌ **Claude Code 响应变慢或卡住**

可能是上下文太长了。试试开启新会话，或者把复杂任务拆成几个小任务分步执行。
